01 What is the Bounty?

Understanding the challenge we set out to solve.

Tinygrad Bounty — Cycle Accurate RDNA3 Emulator

"Add SQTT support to emulator and have it match non-DRAM kernels on real hardware perfectly."

AMD GPUs have a built-in hardware tracing feature called SQTT (Shader Queue Thread Tracing). When enabled, it records the exact clock cycle when each instruction starts executing on the GPU — like a flight recorder for shader programs.

Tinygrad has a software emulator (mockgpu) that simulates GPU execution. The bounty asks us to make this emulator's timing predictions match the real GPU's recorded timestamps perfectly — zero difference — for all instructions that don't touch main memory (DRAM).

Why exclude DRAM? Memory latency is variable and depends on cache state, DRAM controller queues, etc. But compute and LDS (local shared memory) operations have deterministic timing — and that's what we need to nail.

03 The 5 Key Changes

Each precision fix that brought us from "close" to "exact."

LDS

1. LDS Service Cost: 5 → 6 cycles

The LDS (Local Data Share) unit stays busy for 6 cycles per write operation, not 5 as previously modeled.

🏪 Think of it like a checkout lane — each customer (write) takes 6 seconds to process, not 5. If two customers arrive close together, the second waits longer than we thought.
BeforeLDS_SERVICE_CYCLES = 5
Wave 1 waitcnt1 = 37
AfterLDS_SERVICE_CYCLES = 6
Wave 1 waitcnt1 = 38 ✓ (HW = 38)
→ Fixed waitcnt1 delta for both waves
BARRIER

2. Barrier Release Latency: 10 → 6 cycles

After all waves reach a synchronization barrier, it takes 6 cycles to release them (was 10).

🚦 The barrier is like a gate that opens once everyone arrives. We thought the gate mechanism took 10 cycles to open — turns out it's only 6.
BeforeBARRIER_RELEASE_LATENCY = 10
Timing skew in post-barrier instructions
AfterBARRIER_RELEASE_LATENCY = 6
Post-barrier timing distribution matches HW ✓
→ Fixed barrier timing distribution across waves
LDS

3. LDS Write→Read Mode Switch: +1 cycle penalty

When the LDS unit switches from writing to reading, there's a 1-cycle penalty for the direction change.

🚪 Like a revolving door that needs a moment to reverse direction. Going write→write or read→read is free, but switching modes costs 1 extra cycle.
BeforeNo mode switch penalty
Wave 0 waitcnt2 = 31
After+1 cycle on write→read
Wave 0 waitcnt2 = 32 ✓ (HW = 32)
→ Fixed waitcnt2 for Wave 0
SCHEDULING

4. VALU Burst Scheduling

When a wave is executing consecutive math (VALU) instructions, it gets priority to keep going rather than yielding to another wave.

🏃 Like a runner on a track who gets to keep their lane for a burst of sprints, rather than swapping lanes after every step. Prevents unnecessary context switches mid-calculation.
BeforeRound-robin every cycle
v_mov/v_cndmask interleaved wrong
AfterVALU burst priority
v_mov/v_cndmask/v_lshlrev matches HW ✓
→ Fixed VALU instruction distribution between waves
BARRIER

5. Barrier Arrival Penalty: +1 cycle for late wave

The second wave to arrive at a barrier pays 1 extra cycle of overhead.

🚦 Like arriving at a red light — the first car was already there waiting, but you have to come to a stop too. That deceleration costs 1 cycle.
BeforeNo arrival penalty
Wave 1 barrier delta = 1
After+1 for late arrival
Wave 1 barrier delta = 2 ✓ (HW = 2)
→ Fixed Wave 1 s_barrier delta
